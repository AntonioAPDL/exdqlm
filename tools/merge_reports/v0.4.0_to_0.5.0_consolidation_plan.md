### Recommended approach
- Use function-level backend with global-option fallback (consistent with existing option style):
  - Add `backend = c("auto", "R", "cpp")` to builders.
  - `auto` resolves via `getOption("exdqlm.use_cpp_builders", FALSE)` initially.
  - If `cpp` fails and `backend="auto"`, warn and fallback to R; if user explicitly requested `backend="cpp"`, error hard.
- Keep output contract exactly identical to current `seasMod()`/`polytrendMod()` return structure.

### Exact next actions
1. Implement active C++ builders in a new source file (do not revive commented scaffold verbatim).
2. Add thin R wrappers mapping C++ output to current object format + `check_mod()`.
3. Add strict equivalence tests for dimensions and entries across representative parameter grids.
4. Switch default to C++ only after equivalence suite is stable on Linux/macOS/Windows.

Small wiring pattern (recommended):

```r
resolve_builder_backend <- function(backend = c("auto", "R", "cpp")) {
  backend <- match.arg(backend)
  if (backend == "auto") {
    if (isTRUE(getOption("exdqlm.use_cpp_builders", FALSE))) "cpp" else "R"
  } else backend
}

build_with_fallback <- function(cpp_fun, r_fun, backend) {
  b <- resolve_builder_backend(backend)
  if (b == "R") return(r_fun())
  tryCatch(cpp_fun(), error = function(e) {
    if (backend == "cpp") stop(e)
    warning("C++ builder failed; falling back to R: ", conditionMessage(e))
    r_fun()
  })
}
```
